## 1. プロジェクト概要

### 1.1 背景と課題

日本の健康診断（健診）は年に1回以上受診する人が多いが、結果は紙またはPDFで提供され、経年変化を追跡するには手作業での管理が必要となる。また、健診データは極めてセンシティブな個人情報であり、外部サーバーへの送信に対する心理的障壁が大きい。

本プロジェクトは、これらの課題を「完全ローカル処理」というアプローチで解決する。

### 1.2 プロダクトビジョン

「スマホで健診結果を撮影するだけで、自分の健康推移が一目でわかる。データは端末の外に出ない。」

### 1.3 スコープ境界

| スコープ内 | スコープ外 |
|---|---|
| オンデバイスOCRによる健診結果読み取り | 健診予約・受診管理 |
| ローカルDBでのデータ永続化 | 医療機関との電子データ連携 (HL7/FHIR) |
| 時系列チャートによる可視化 | AIによる健康アドバイス生成 |
| 共有機能 (PDF/画像エクスポート) | 他ユーザーとのリアルタイムデータ同期 |
| バイオメトリクス/PINによるアプリロック | ウェアラブルデバイス連携 |

## 3. 機能要件

### 3.1 OCRエンジン

#### 3.1.1 技術選定

| 項目 | 仕様 |
|---|---|
| ライブラリ | google_mlkit_text_recognition |
| モデル | Japanese + Latin on-device model |
| 外部通信 | なし（初回モデルDL後は完全オフライン） |
| 入力ソース | カメラリアルタイム撮影 / ギャラリー選択 |
| 前処理 | 自動クロッピング、過度補正、グレースケール変換 |

#### 3.1.2 テキストパーサー

OCRで得られた生テキストから構造化データを抽出するパーサー。本アプリの技術的コア。

- **抽出対象:** 検査項目名、数値、単位、基準範囲、判定フラグ (A/B/C/D または H/L)
- **パース戦略:** 表形式検出 → 行単位解析 → 正規表現マッチング
- **フォーマット対応:** 初期リリースでは主要3パターンに対応（後述）。プラグイン方式で拡張可能な設計とする
- **信頼度スコア:** 各フィールドに0.0〜1.0の信頼度を付与し、低信頼度のフィールドはハイライトしてユーザーに確認を促す

#### 3.1.3 対応フォーマット (Phase 1)

| パターン | 説明 | 例 |
|---|---|---|
| タイプA | 縦型表形式（検査項目が左列、数値が右列） | 大手企業健診結果 |
| タイプB | 横型表形式（検査項目がヘッダー行、値が下行） | 人間ドック型 |
| タイプC | フリーテキスト（表罫線なし、ラベルと値が並列） | 自治体特定健診 |

### 3.2 データモデル

#### 3.2.1 エンティティ一覧

| エンティティ | 説明 | 主要属性 |
|---|---|---|
| Profile | 利用者プロフィール | id, name, birthDate, sex |
| Checkup | 健診受診記録 | id, profileId, date, facilityName, sourceImagePath |
| TestResult | 個別検査結果 | id, checkupId, itemCode, itemName, value, unit, refLow, refHigh, flag |
| TestCategory | 検査カテゴリマスタ | id, name, displayOrder |
| TestItemMaster | 検査項目マスタ | id, categoryId, standardName, aliases[], unit, refLow, refHigh |

#### 3.2.2 検査カテゴリ体系

| カテゴリ | 主な検査項目 |
|---|---|
| 身体計測 | 身長, 体重, BMI, 腹囲 |
| 血圧 | 収縮期血圧, 拡張期血圧 |
| 血液一般 | 赤血球, 白血球, 血小板, ヘモグロビン, ヘマトクリット |
| 脂質 | LDL-C, HDL-C, 中性脂肪, 総コレステロール |
| 糖代謝 | 空腹時血糖, HbA1c |
| 肝機能 | AST(GOT), ALT(GPT), γ-GTP, ALP |
| 腎機能 | クレアチニン, eGFR, BUN, 尿酸 |
| 尿検査 | 尿糖, 尿蛋白, 尿潜血 |

#### 3.2.3 データ保存戦略

| 項目 | 仕様 |
|---|---|
| ローカルDB | Isar（後続検討: sqflite, Drift） |
| 画像保存 | アプリ内部ストレージ (getApplicationDocumentsDirectory) |
| バックアップ | ローカルファイルエクスポート (JSON) |
| 暗号化 | DBレベルでの暗号化を検討 (flutter_secure_storage for keys) |

### 3.3 可視化機能

#### 3.3.1 ダッシュボード

- 最新健診結果のサマリーカード表示
- カテゴリ別の信号色表示（緑: 正常 / 黄: 要注意 / 赤: 異常）
- 前回との変化矢印 (↑↓→) 付き
- 次回健診までのカウントダウン（任意設定）

#### 3.3.2 チャート種別

| チャート | 用途 | ライブラリ |
|---|---|---|
| 折れ線グラフ | 個別項目の経年推移（基準範囲バンド付き） | fl_chart |
| レーダーチャート | カテゴリ別総合バランス表示 | fl_chart |
| 棒グラフ | 複数項目の比較表示 | fl_chart |
| ヒートマップ | 全項目の経年正常/異常マトリクス | カスタム描画 |

### 3.4 共有機能 (Share)

プライバシーファーストの原則に基づき、共有は常にユーザーの明示的な操作によってのみ発生する。

#### 3.4.1 共有手段

| 手段 | 説明 | 優先度 |
|---|---|---|
| PDFレポート | 選択期間の結果を一覧表+チャート付きPDFとして生成。OSのShare Sheet経由で送信 | Must |
| チャート画像 | 表示中のチャートをPNGとしてキャプチャし、Share Sheet経由で共有 | Must |
| JSONエクスポート | 構造化データをJSONでエクスポート。他端末への移行やバックアップにも利用 | Should |
| CSVエクスポート | Excel等で開ける形式でのデータ共有 | Could |

#### 3.4.2 共有時のプライバシー保護

- 共有前に必ず確認ダイアログを表示（「以下のデータを共有します」）
- 共有対象の項目をユーザーが選択可能（全項目 or 選択項目）
- 個人名・生年月日の包含/除外をオプションで制御
- 共有履歴をログとしてローカル保存

## 4. 非機能要件

### 4.1 プライバシー・セキュリティ

| 要件 | 説明 |
|---|---|
| 完全ローカル処理 | OCR処理、データ保存、可視化の全てが端末内で完結すること |
| ネットワーク通信禁止 | アプリからの外部サーバーへの通信は一切行わない（ML KitモデルDLを除く） |
| アプリロック | バイオメトリクス (Face ID / Touch ID / 指紋) またはPINコードによるアクセス制御 |
| データ暗号化 | DBファイルの暗号化を検討。最低限、キーをflutter_secure_storageで管理 |
| アナリティクス禁止 | 利用統計・行動追跡・広告トラッキング等の外部送信は一切行わない |

### 4.2 パフォーマンス

| 指標 | 目標値 |
|---|---|
| OCR処理時間 | 1ページあたり3秒以内 (ミッドレンジ端末) |
| アプリ起動時間 | 2秒以内 (コールドスタート) |
| チャート描画 | 60fps でのスムーズなアニメーション |
| ローカルDBクエリ | 100件の健診記録 × 50項目で100ms以内 |
| アプリサイズ | 初回インストール50MB以内 (モデル含む) |

### 4.3 対応プラットフォーム

| プラットフォーム | 最低バージョン |
|---|---|
| iOS | 16.0+ |
| Android | API Level 26+ (Android 8.0 Oreo) |

### 4.4 アクセシビリティ

- ダイナミックタイプ対応（テキストサイズ調整）
- ダークモード対応
- VoiceOver / TalkBack での基本操作を保証
- 色覚異常対応（信号色にアイコン/パターンを併用）

## 8. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| OCR精度が低い健診結果がある | 高 | 必ずOCR結果確認・修正UIを経由させる。信頼度スコアで要確認箇所を明示 |
| 検査項目名の表記揺れが大きい | 高 | TestItemMasterにエイリアス配列を持たせ、ファジーマッチングで対応 |
| 健診フォーマットが想定外 | 中 | 手動入力をフォールバックとして常に提供。対応フォーマットを段階的に追加 |
| ML Kit日本語モデルの精度限界 | 中 | 画像前処理（二値化・コントラスト強調）で補完 |
| 大量データ時のパフォーマンス低下 | 低 | Isarのインデックス設計とクエリ最適化 |

## 9. 成功指標 (KPI)

| 指標 | 目標 | 計測方法 |
|---|---|---|
| OCR読み取り成功率 | 数値フィールド90%以上正確 | テストデータセット (50枚以上) で検証 |
| ユーザー修正率 | 全フィールドの20%以下 | OCR結果確認画面での修正操作ログ |
| スキャン完了率 | スキャン開始から保存までの完遂率80%以上 | ローカルファンネル計測 |
| チャート閲覧頻度 | 週1回以上のダッシュボード表示 | ローカル利用統計 (外部送信なし) |
